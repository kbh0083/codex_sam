- 새 프롬프트 파일을 추가했다.
  - `/Users/bhkim/Documents/codex_prj_sam_asset/app/prompts/IBK.txt`
- 매핑을 추가했다.
  - `/Users/bhkim/Documents/codex_prj_sam_asset/app/prompts/counterparty_prompt_map.yaml`
- 핵심 규칙은 아래로 고정했다.
  - `base_date`는 메일 `Date`가 아니라 문서 본문 `기준일자`
  - `운용사`에 `삼성` 포함 row만 추출
  - 주문 근거는 `정산액` + `예정 정산액 기준일+N` net column만 사용
  - `설정액`/`해지액`은 주문 근거로 사용하지 않음
  - `정산액`은 내부 `CONFIRMED`
  - `예정 정산액 기준일+N`은 내부 `PENDING`
  - downstream는 `only_pending=true`지만, 추출 단계는 full internal order set 유지
- review finding 반영:
  - future-only row에서도 `설정액`/`해지액` 기반 split order가 생기지 않도록 net-column-only 규칙을 강하게 적었다.
- 실제 기대 internal 결과는 3건으로 고정했다.
  - `BBCA00 / 2026-04-08 / CONFIRMED / RED / t_day=0 / -54,788,126`
  - `BBCA00 / 2026-04-08 / PENDING / RED / t_day=1 / -95,003,244`
  - `BBCA00 / 2026-04-08 / PENDING / SUB / t_day=2 / 2,794,977`
- `only_pending=true` 최종 출력 기대값도 테스트로 고정했다.
  - 최종 1건
  - `settle_class="1"`
  - `order_type="1"`
  - `t_day="01"`
  - `transfer_amount="54,788,126"`

### 3. 카디프 프롬프트 재검수와 계약 수정
- 사용자 리뷰 포인트는 두 가지였다.
  - 프롬프트가 모델에 보이지 않는 file-name cue에 의존하지 않아야 함
  - live regression이 subset 검증이 아니라 전체 exact compare여야 함
- 그 결과 카디프 프롬프트를 아래처럼 정리했다.
  - attachment precedence는 loader-visible marker만 사용
    - PDF cue: `## Page`, `[PAGE]`
    - XLSX cue: `## Sheet`, `[SHEET]`
  - spreadsheet copy는 `duplicate XLSX copy; use PDF attachment`로 즉시 skip
  - 내부 추출은 `CONFIRMED`
  - `only_pending=true`는 downstream 후처리 계약으로만 설명
- 특히 아래 오해를 바로잡았다.
  - 잘못된 과거 상태:
    - 카디프 프롬프트에 “every emitted order is `PENDING`”
    - `app/output_contract.py`에서 카디프 `settle_class`를 무조건 `"1"`로 rewrite
  - 현재 고친 상태:
    - 카디프 프롬프트는 internal `CONFIRMED`
    - final `PENDING`는 `only_pending=true` 경로에서만 발생
    - `app/output_contract.py`는 카디프에 대해 sort만 적용하고 `settle_class`를 강제 rewrite하지 않음
- 카드िफ live regression도 바꿨다.
  - 실제 fixture:
    - `/Users/bhkim/Documents/codex_prj_sam_asset/output/correct_result/11_카디프__카디프_251127.pdf`
  - baseline:
    - `/Users/bhkim/Documents/codex_prj_sam_asset/output/correct_result/11_카디프.json`
  - `DocumentExtractionRequest(..., use_counterparty_prompt=True, only_pending=True)`로 실행
  - `status`, `reason`, `base_date`, `issues`, `orders`를 baseline과 exact compare
  - fixture 또는 baseline이 없으면 synthetic fallback이 아니라 `skip`

### 4. IBK / 흥국생명-hanais only_pending 재검수
- 사용자 지적대로 카디프에서 `only_pending` 의미를 잘못 섞은 전례가 있었기 때문에, IBK와 HANAIS도 다시 점검했다.
- 결론:
  - `IBK` 프롬프트는 괜찮았다.
    - internal extraction semantics와 downstream `only_pending=true`를 분리해서 적고 있었다.
  - `흥국생명-hanais`는 `only_pending` 의미 혼동은 없었지만, attachment precedence에 file-name cue가 남아 있었다.
    - 이번 세션에서 그 cue를 제거하고 loader-visible marker 기준으로 정리했다.
    - internal `CONFIRMED` extraction과 downstream `only_pending=false` 분리는 그대로 유지된다.
- 다만 문서상 설명만 있고 회귀 테스트가 약해서, 아래를 추가했다.
  - `IBK` guidance에 `only_pending=true`, `full internal order set` 문구가 들어 있는지 검증
  - `흥국생명-hanais` guidance에 `only_pending=false`, `keep confirmed orders as confirmed` 문구가 들어 있는지 검증
  - `IBK` live regression에 final `only_pending=true` output 1건 검증 추가
  - `흥국생명-hanais` live regression에 final `only_pending=false` confirmed 2건 검증 추가
