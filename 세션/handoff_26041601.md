## 시행착오 / 반드시 기억할 문제들

### 1. “소수점 이하 제거”를 blanket truncation으로 해석하면 안 됨
- 처음 사용자 요구만 보면 모든 decimal을 버리는 것처럼 읽히지만, 이걸 그대로 구현하면 KB 같은 실제 decimal 문서가 깨진다.
- 이번 세션의 정답은:
  - **artifact만 제거**
  - **true decimal은 보존**

### 2. 중복 제거를 단순 `fund_code` 기준으로 하면 정상 주문을 잃는다
- 같은 펀드라도 `SUB/RED`, `T0/T+1`, `CONFIRMED/PENDING`이 같이 있을 수 있다.
- canonical 7-key 기준으로 dedupe 해야 한다.

### 3. generic `펀드` fallback은 explicit `운용사코드`보다 뒤에 와야 한다
- KB 때문에 generic `펀드` fallback이 필요하지만,
- mixed-header 문서에서는 explicit `운용사코드`가 반드시 우선해야 한다.
- 이 precedence를 잘못 두면 삼성 scope row가 통째로 누락된다.

### 4. canonical amount는 dedupe만이 아니라 table hint key에도 필요하다
- 처음에는 amount canonicalization을 dedupe/signature 쪽만 고쳤는데,
- 나중에 review에서 markdown table order-type hint는 여전히 raw amount key를 쓰는 문제를 발견했다.
- 즉 canonical amount는 **hint key / signature / final output** 전체에서 일관돼야 한다.

### 5. merge report도 코드처럼 review 대상이다
- 보고서가 실제 증빙 파일과 안 맞으면 “PASS 근거”가 재현 불가능해진다.
- 이번에는 helper rerun 결과가 바뀌었는데 보고서가 old preflight evidence를 계속 가리키는 문제가 있었다.
- 또 finding 번호를 review thread와 다르게 적으면 traceability가 깨진다.

### 6. 문서는 “말미 업데이트 메모”만 붙이면 안 끝난다
- 처음에 2026-04-16 메모만 덧붙였더니,
  - 상단은 여전히 `최신 상태 (2026-04-14)`
  - 본문은 여전히 `6단계`
  - prompt 설명도 구문 그대로
- 결국 문서 전체가 서로 충돌하는 상태가 됐다.
- 다음에 문서 업데이트를 할 때는
  - 상단 최신 기준선
  - 본문 핵심 기술 설명
  - stable history 구분
를 같이 봐야 한다.

### 7. acceptance 로그가 있다고 해서 source file도 현재 남아 있다고 가정하면 안 된다
- 이번 handoff 최종 검수에서 새로 잡은 문제다.
- `output/debug/was_merge_20260416_111730/preflight/helper_pytest*.log`에는 분명 `tests/test_variable_annuity_extract_helpers.py` 실행 기록이 남아 있다.
- 하지만 현재 `/Users/bhkim/10_project/01_samsung_asset/samsung_ai_portal_backend/tests/` 아래에서는 해당 source file 존재를 확인하지 못했다.
- 즉,
  - 과거 acceptance 로그
  - 현재 tree의 실제 파일 상태
  - 문서에 적힌 경로
  를 분리해서 검증해야 한다.
- 다음 세션에서 helper pytest를 다시 돌릴 때는 source file 존재부터 먼저 확인해야 한다.

### 8. `hanis` vs `hanais`는 실제 운영 데이터 차이다
- DB row/company_name 쪽에는 `흥국생명-hanis`
- local prompt file/case naming 쪽에는 `흥국생명-hanais`
- 이것 때문에 단순 grep 결과만 보고 “오타니까 통일하자”라고 건드리면 운영 연계가 꼬일 수 있다.

### 9. full live suite보다 deterministic/unit + targeted rerun이 더 신뢰할 수 있는 구간이 있다
- 특히 LLM 연결 noise가 있으면 full suite 한 번 실패한 결과를 제품 regression으로 오판하기 쉽다.
- 이번 세션도 실제로는 deterministic/unit + impacted case rerun + pipeline smoke 조합으로 acceptance를 닫았다.
