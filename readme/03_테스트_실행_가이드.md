## 판정 기준
- `PASS`
  - 정답지 JSON과 최종 payload가 `base_date`, `status`, `reason`, `issues`, `orders` 기준으로 일치해야 한다.
  - 원본 지시서 대조에서 주문, 금액, 날짜, 방향, 펀드 정보에 부정확성이 없어야 한다.
  - 즉 DB 저장용 최종 데이터로 승인 가능한 경우에만 `PASS`다.
- `FAIL`
  - 정답지 mismatch, 원본 대조 mismatch, 주문/금액/날짜/방향/펀드 정보 중 하나라도 부정확하면 무조건 `FAIL`이다.
  - `source review PASS`, `semantic match`, `accepted review`만으로 최종 `PASS`를 주지 않는다.
- `BLOCKED`
  - 정답지가 없거나, preflight/연결/실행 환경 문제로 판정을 시작할 수 없을 때 사용한다.
- `SMOKE_PASS`
  - 구조, 파이프라인, 연결 확인용 비정확도 테스트에만 제한적으로 사용한다.
  - 지시서 데이터 추출 성공 판정에는 사용하지 않는다.

## 추출 원칙
- 원본 지시서가 최종 source of truth다.
- 정답지, 프롬프트, 과거 accepted 결과는 보조 기준일 뿐이며, 원본 지시서와 충돌하면 원본 기준으로 수정한다.
- 추출은 prompt-first를 원칙으로 한다.
  - 특히 `base_date`, `t_day`, `order_type`처럼 문맥 해석이 필요한 필드는 LLM 판단이 먼저 개입할 수 있어야 한다.
- deterministic fallback은 권위 추출 경로가 아니다.
  - deterministic 결과가 정확도를 높인다는 근거가 있을 때만 사용한다.
  - 정확도를 해치거나 LLM 판단을 가로막으면 제거하거나 최후 fallback으로 강등한다.
- EML의 wrapper metadata(`Date (Asia/Seoul)` 등)는 attached 지시서 본문 날짜보다 우선하지 않는다.
  - 본문 표, 제목, 반복 header에 더 강한 날짜 근거가 있으면 그 날짜를 최종 기준일로 사용한다.
- duplicate/legacy copy 성격 문서는 extractor core precheck 통과 여부로 판정하지 않는다.
  - 현재 기준 판정은 최종 `status/reason`이 상위 실행 경계(local service/component/handler, WAS task/pipeline)에서 올바르게 `SKIPPED`로 surface되는지 본다.

## 거래처별 `only_pending` 옵션
- `only_pending`은 추출 규칙이 아니라 최종 출력 직전 downstream post-processing 계약이다.
- 내부 추출은 항상 원문 기준의 full internal order set을 먼저 만든 뒤, `only_pending=true`인 경우에만 최종 JSON/CSV 직전에 후처리를 적용한다.
- `only_pending=true`일 때 적용 규칙:
  - 기존 `settle_class == PENDING` 주문을 제거한다.
  - 남은 주문의 `settle_class`를 내부적으로 `PENDING`으로 맞춘 뒤 외부 출력 코드 `"1"`로 저장한다.
- `only_pending=false`일 때는 내부 추출 결과를 그대로 직렬화한다.
- 별도 거래처별 output contract도 최종 JSON/CSV 직전에만 적용한다.
  - 2026-04-24 로컬 기준 `동양생명`, `한화생명`, `신한라이프`는 최종 직렬화 row 중 `t_day="02"` 주문을 저장 payload에서 제거한다.
- 거래처 prompt가 `[[COUNTERPARTY_PROMPT_META]]` / `[[/COUNTERPARTY_PROMPT_META]]` structured marker를 가지면, 해당 metadata의 `fixed_stage_columns`가 `t_day`와 `transfer_amount` 추출 대상 컬럼을 고정한다.
  - marker가 없거나 `use_counterparty_prompt=false`면 기존 generic net-preference 규칙을 그대로 사용한다.
- 2026-04-24 로컬 기준 fixed-column 거래처 규칙:
  - `동양생명`: `정산내역` same-day는 `정산액`을 제외하고 `설정액`/`해지액`만 추출한다. `예상내역`은 기존 future `설정금액`/`해지금액` 규칙을 유지한다.
  - `IBK`: same-day는 `정산액`을 제외하고 `설정액`/`해지액`만 추출한다. future `예정 정산액 기준일+1/+2/+3`는 그대로 future evidence로 유지한다.
- source of truth:
  - WAS/운영 기준은 DB row의 `only_pending` 값이다.
  - 로컬 테스트/정답지 기준은 현재 검증된 prompt, 테스트, 보고서, 정답지 세트와 일치해야 한다.
  - 파일명이나 문서 내용만 보고 `only_pending` 값을 새로 추정하지 않는다.
- 테스트와 정답지 작성 규칙:
  - `case_manifest.json`, `validation_summary.json`, 보고서 표에는 케이스별 `only_pending=true/false`를 명시한다.
  - 정답지 JSON/CSV는 내부 intermediate 결과가 아니라 `only_pending`까지 반영된 최종 DB 저장 payload를 기준으로 작성한다.
  - 내부 추출 결과와 최종 저장 payload를 혼동하지 않는다.
- 현재 거래처별 옵션:
  - `only_pending=false`: `동양생명`, `신한라이프`, `한화생명`, `흥국`, `KDB`
  - 위 목록을 제외한 나머지 거래처는 `only_pending=true`로 본다.
  - 세부 문서/케이스 적용값은 항상 최신 WAS DB row 또는 최신 검증된 answer set/보고서와 함께 다시 확인한다.

## 정답지 규칙
- 정답지 경로는 아래를 따른다.

- 파일 세트는 아래 3종을 반드시 함께 둔다.
  - `<exact_document_stem><original_ext>`

  - `<exact_document_stem>.csv`

  - source copy: `<exact_document_name>`
  - answer json: `<exact_document_name>.json`
  - answer csv: `<exact_document_name>.csv`
- 이 예외에서도 alias 이름은 금지하고, 실제 `document` 파일명 전체를 그대로 사용한다.

- 같은 업무 지시처럼 보여도 `xlsx`, `pdf`, `eml`은 문서별로 정답 세트를 각각 둔다.

## 참고
- 과거 `output/debug/...`, `test_report/*.md` 경로는 legacy 기록이다.
