이 문서는 현재 추출 엔진에서 반복적으로 발생하는 아래 문제를 줄이기 위한  
**1차 설계안**을 정리한 문서다.

- 특정 문서 이슈를 해결하려고 공통 엔진을 수정하면
- 다음 대표 문서 배치에서 다른 문서가 다시 흔들리고
- `fallback / coverage / 후처리` 쪽 패치가 계속 누적되는 문제

  1차 수준에서 명확히 정리하는 것

중요:

- 이 문서는 **설계 문서**다.
- 이 문서는 **장기 구조 개편 제안서**이지, 현재 코드가 이미 이 구조로 동작한다는 뜻은 아니다.

## 0. 왜 이 설계가 필요한가

최근 검수 과정에서 반복적으로 드러난 패턴은 아래와 같다.

1. 대표 문서 배치에서 특정 거래처 문서가 실패한다.
2. [app/extractor.py](/Users/bhkim/Documents/codex_prj_sam_asset/app/extractor.py)의
   `fallback / coverage / 후처리` 로직을 수정한다.
3. 수정된 공통 로직이 전체 문서에 동시에 적용된다.
4. 다음 배치에서 다른 문서가 새로 흔들린다.

이 패턴은 코드 품질이 낮아서라기보다,  
**거래처별 차이를 공통 엔진에 계속 누적하는 방식 자체가 구조적으로 불안정**하기 때문에 발생한다.

현재 구조에서는 아래 문제가 같이 존재한다.

- 거래처 프롬프트는 LLM 해석에만 영향을 준다.
- 실제 실패는 자주 `fallback / coverage / 후처리`에서 발생한다.
- 따라서 프롬프트만으로는 해결되지 않는 문서가 반복적으로 나온다.
- 그 결과 공통 엔진에 문서별 예외 패치가 쌓인다.

결론적으로, 앞으로는

- 공통 엔진은 더 보수적으로 고정하고
- 거래처별 차이는 **DB 기반 구조화 프로필**로 옮기고
- `fallback / coverage / 후처리`도 그 프로필을 읽게 만드는 방향이 필요하다.

## 1. 프리징을 위한 현실적인 기준안

현재처럼 모든 known 거래처 문서에 대해  
`use_counterparty_prompt=false` 와 `true`를 모두 동일 품질로 계속 맞추려 하면
공통 엔진 패치가 끝없이 늘어날 가능성이 크다.

그래서 프리징 기준은 아래처럼 **운영 경로 기준**으로 나누는 것이 현실적이다.

### 1-1. 운영 기준

- `known counterparty`
  - 정식 운영 경로: `use_counterparty_prompt=true`
  - 거래처 프롬프트 + 거래처 프로필 적용 결과를 기준으로 본다.
- `unknown counterparty`
  - 정식 운영 경로: `use_counterparty_prompt=false`
  - 일반 엔진 + 안전 fallback 결과를 기준으로 본다.

### 1-2. 프리징 통과 조건

- known 거래처 대표 문서는 `use_counterparty_prompt=false` 와 `true` 두 경로 모두 정상 추출되어야 한다.
- unknown 거래처 대표 문서는 `use_counterparty_prompt=false` 경로에서 정상 추출되어야 한다.
- known 거래처의 `false` 경로도 아래 조건을 모두 만족해야 한다.
  - 비정상 종료 없음
  - 저장 차단 없음
  - 명백한 오추출 없음
  - 최종 결과가 원문 검수 기준과 일치

즉 현재 프리징 기준은 아래처럼 유지한다.

- known 거래처: `false` / `true` 모두 통과
- unknown 거래처: `false` 통과

다만 장기 운영 관점에서는 아래를 별도 참고안으로 둘 수 있다.

- 운영 모니터링과 SLA는 known 거래처의 `true` 경로를 더 우선적으로 본다.
- 하지만 프리징 완료 판단 자체를 `true`만으로 내리지는 않는다.

이 구분이 필요한 이유는,
known 거래처라도 실제로는 같은 문서에 대해

- generic engine 경로
- profile-aware 경로

두 가지가 모두 살아 있기 때문이다.

따라서 최소한 프리징 시점까지는 두 경로를 모두 정상 상태로 맞춰 두는 것이 맞다.

### 1-3. 공통 엔진 수정 허용 기준

공통 엔진 수정은 아래 경우에만 허용하는 것이 좋다.

- `false`와 `true` 모두에서 재현되는 문제
- unknown 거래처 경로에서 재현되는 문제
- 저장 차단
- `ORDER_COVERAGE_MISMATCH`
- 후처리 오판
- stale issue 정리 실패

반대로 아래는 공통 엔진에 직접 넣지 않는 것이 좋다.

- 특정 거래처만 쓰는 섹션 제목 규칙
- 특정 거래처만 쓰는 금액 컬럼 우선순위
- 특정 거래처만 통하는 hybrid header 해석
- 특정 거래처 양식만을 위한 예외성 fallback

이런 규칙은 DB 프로필로 빼야 한다.

## 2. DB에 추가해야 할 거래처 프로필 필드

WAS에서는 이미 거래처 관리 테이블이 있고,
이메일 제목 + 발신 도메인으로 거래처를 식별해 프롬프트를 가져오는 구조를 계획하고 있다.

장기적으로는 여기서 프롬프트만이 아니라
**코드가 읽을 수 있는 거래처 프로필**도 함께 관리해야 한다.

기준 모델 파일:

- [variable_counterparty.py](/Users/bhkim/10_project/01_samsung_asset/samsung_ai_portal_backend/src/app/services/variable_annuity/models/variable_counterparty.py)

### 2-1. 필수 필드

- `use_counterparty_prompt_default: bool`
- `counterparty_prompt_common: text`
- `counterparty_prompt_overrides_json: json`
- `use_counterparty_profile_default: bool`
- `counterparty_profile_json: json`
- `active: bool`
- `profile_version: int`
- `updated_by: string`
- `updated_at: datetime`

### 2-2. 권장 필드

- `title_match_keywords: json`
- `sender_domains: json`
- `document_family_rules_json: json`
- `notes: text`
- `last_validated_at: datetime`
- `last_validated_result: json`

### 2-3. 필드 의미

- `use_counterparty_prompt_default`
  - 거래처 전체의 기본 프롬프트 활성화 여부
  - 실제 적용은 `document_profiles[*].enabled.use_counterparty_prompt`가 우선하고, 없을 때만 이 기본값을 사용
- `counterparty_prompt_common`
  - 거래처 전체에 공통으로 적용할 수 있는 기본 프롬프트
- `counterparty_prompt_overrides_json`
  - `doc_type` 또는 `document_family`별 추가 guidance
  - 최종 guidance는 `counterparty_prompt_common + prompt override`로 조합
- `use_counterparty_profile_default`
  - 거래처 전체의 기본 프로필 활성화 여부
  - 실제 적용은 `document_profiles[*].enabled.use_counterparty_profile`가 우선하고, 없을 때만 이 기본값을 사용
- `counterparty_profile_json`
  - 코드가 직접 읽는 구조화 규칙
- `document_family_rules_json`
  - `document_family`를 결정하기 위한 우선 매칭 규칙
  - 제목, 발신 도메인, 파일명, `doc_type`, loader가 뽑은 title/header 후보 등을 입력으로 사용
- `profile_version`
  - 변경 이력 및 회귀 검수 기준점 관리용

### 2-4. 공통 우선순위 규칙

`prompt / profile / output policy / enabled flag`는 subsystem마다 따로 우선순위를 해석하면 안 된다.
1차 설계에서는 아래 공통 규칙으로 고정한다.

1. `document_family` override
2. `doc_type` profile
3. 거래처 top-level default

즉 아래 항목은 모두 같은 precedence를 따라야 한다.

- prompt override
- `use_counterparty_prompt` 유효값
- `use_counterparty_profile` 유효값
- fallback strategy
- validation rule
- output policy

예를 들어:

- `document_family`에 `cardif_sectioned_pdf` override가 있으면 그것을 먼저 사용
- 해당 override가 없으면 `pdf` profile을 사용
- 둘 다 없으면 top-level default를 사용

이 규칙은 registry, prompt renderer, reconcile, fallback 전략 선택 모두에 동일하게 적용한다.

## 3. 거래처 프로필 JSON 1차 스키마

1차 구현에서는 거래처 row 하나에 단일 `fallback_strategy`를 두기보다,
**형식 또는 문서군별 하위 프로필**을 두는 구조가 더 안전하다.

즉 아래처럼

- 거래처 공통 메타데이터
- `document_profiles`
  - `pdf`
  - `xlsx`
  - `eml`
  - `html`
  - 또는 `document_family`

로 나누는 형태가 적절하다.

### 3-2. `counterparty_prompt_overrides_json` 예시

```json
{
  "document_profiles": {
    "pdf": {
      "prompt_override": "영문 섹션형 문서에서는 Subscription/Redemption 제목을 방향 판단의 우선 근거로 사용한다."
    },
    "xlsx": {
      "prompt_override": "결제일이 명시된 경우 제목 근처 날짜보다 결제일 컬럼을 base_date로 우선 사용한다."
    }
  }
}
```

### 3-3. `document_family_rules_json` 예시

```json
{
  "rules": [
    {
      "document_family": "cardif_sectioned_pdf",
      "doc_type": "pdf",
      "title_keywords": ["Subscription", "Redemption"],
      "header_keywords": ["NAV Amount(KRW)"]
    },
    {
      "document_family": "hanhwa_schedule_html",
      "doc_type": "html",
      "title_keywords": ["이체예상금액", "펀드계"]
    }
  ]
}
```

핵심 원칙은 아래 네 가지다.

- `counterparty_prompt_common`과 `counterparty_prompt_overrides_json`은 LLM용 설명이다.
- `counterparty_profile_json`은 코드용 규칙이다.
- `only_pending` 기본 정책도 거래처 단일값이 아니라 `document_profiles[*].output_policy`에서 읽는다.
- 활성화 여부도 거래처 단일값이 아니라 `document_profiles[*].enabled`가 우선한다.

그리고 이 구조는 아래처럼 나누어 가져가는 것이 안전하다.

- 공통 프롬프트
- 형식/문서군별 prompt override
- `doc_type` 또는 `document_family`별 코드 규칙
- `doc_type` 또는 `document_family`별 output policy
- `doc_type` 또는 `document_family`별 enabled policy

즉 LLM용 설명과 코드용 규칙은 분리하되,
둘 다 형식 또는 문서군 단위로 세분화할 수 있어야 한다.

### 3-4. `document_family` 결정 책임과 우선순위

`document_family`는 extractor가 즉석에서 추론하면 안 된다.
이 값은 **CounterpartyRegistry가 결정**하고, extractor에는 최종 결과만 전달하는 것이 맞다.

`document_family`는 1회에 완전히 확정되지 않아도 된다.
1차 설계에서는 아래처럼 **2단계 확정**으로 본다.

- 1차 family
  - 이메일 제목, 발신 도메인, 파일명, `doc_type`만으로 결정 가능한 값
- 2차 family
  - `DocumentLoader`가 뽑은 title/header 후보까지 포함해 보강한 최종 값

결정 순서는 아래처럼 고정한다.

1. 상위 호출자나 DB에서 명시적으로 전달된 `document_family`
2. `document_family_rules_json`을 이용한 CounterpartyRegistry의 1차 매칭
   - 입력: 이메일 제목, 발신 도메인, 파일명, `doc_type`
3. `DocumentLoader` 이후 CounterpartyRegistry의 2차 재평가
   - 입력: 이메일 제목, 발신 도메인, 파일명, `doc_type`, loader가 뽑은 title/header 후보
4. 그래도 못 정하면 `doc_type`

즉 책임선은 아래와 같다.

- `DocumentLoader`
  - `doc_type`와 title/header 후보 같은 분류 입력값만 제공
- `CounterpartyRegistry`
  - 1차 family 결정
  - loader 결과를 반영한 2차 family 최종 결정
- `FundOrderExtractor`
  - 이미 결정된 `document_family`를 소비만 함

이 순서를 문서 수준에서 먼저 못박아 두면,
나중에 registry / loader / extractor 중 어디엔가 ad hoc family classifier가 생기는 것을 줄일 수 있다.

## 4. 코드에서 어디를 어떻게 분리해야 하는가

## 6. WAS 반영 방향

WAS에서는 아래 흐름으로 가는 것이 가장 자연스럽다.

1. 이메일 제목 + 발신 도메인으로 거래처 row 조회
2. 거래처 row에서 아래 값을 읽는다.
   - `use_counterparty_prompt_default`
   - `counterparty_prompt_common`
   - `counterparty_prompt_overrides_json`
   - `use_counterparty_profile_default`
   - `counterparty_profile_json`
   - `document_family_rules_json`
3. CounterpartyRegistry가 이메일 메타데이터와 파일명, `doc_type` 기준으로 **1차 CounterpartyConfig**를 만든다
4. `DocumentLoader`가 문서를 읽고 `doc_type`, title/header 후보를 뽑는다
5. CounterpartyRegistry가 loader 힌트를 반영해 **최종 CounterpartyConfig**를 확정한다
6. extractor는 최종 `CounterpartyConfig`만 받아 처리

즉 long term 구조는 아래처럼 정리된다.

```text
이메일 메타데이터
  -> 거래처 조회(DB)
  -> DocumentLoader
  -> 1차 CounterpartyConfig 생성
  -> loader 힌트 반영
  -> 최종 CounterpartyConfig 확정
  -> FundOrderExtractor
  -> fallback / coverage / 후처리 (profile-aware)
  -> 최종 결과
```

## 7. 1차 도입 순서

가장 안전한 도입 순서는 아래다.

1. `CounterpartyConfig` / `CounterpartyRegistry` 계층 추가
2. WAS에서 `use_counterparty_prompt_default`, `counterparty_prompt_common`, `counterparty_prompt_overrides_json`를 먼저 DB 연동
3. `use_counterparty_profile_default`, `counterparty_profile_json`, `document_family_rules_json` 스키마 추가
4. CounterpartyRegistry에 1차/2차 `document_family` 결정 로직 추가
5. prompt/profile/output policy 공통 precedence를 코드 상수 또는 helper로 고정
6. fallback 전략을 profile 기반으로 분리
7. coverage / 후처리 / output policy를 profile 기반으로 읽게 변경
8. 대표 문서 골든셋 회귀 테스트 고정

이 순서를 따르면

- 기존 동작을 크게 깨지 않고
- 단계적으로 profile-aware 구조로 이동할 수 있다.

## 9. 결론

이번 설계의 핵심은 아래 한 문장으로 요약할 수 있다.

**공통 엔진을 계속 문서별 예외로 패치하는 방식에서 벗어나, 거래처별 차이를 DB의 구조화 프로필로 분리하고 공통 엔진은 보수적으로 고정한다.**

즉 앞으로의 방향은 아래와 같다.

- 프리징 기준은 운영 경로 기준으로 나눈다.
- 거래처 프롬프트는 LLM용 설명으로 유지한다.
- 거래처 프로필은 코드가 읽는 구조화 규칙으로 분리한다.
- `fallback / coverage / 후처리`도 거래처 프로필을 읽도록 바꾼다.

이 문서는 이후 구현 일정이 잡혔을 때,
실제 작업 범위를 자르기 위한 1차 기준 문서로 사용한다.
