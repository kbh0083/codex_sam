## 1. 이 프로젝트가 하는 일

이 프로젝트는 보험사/운용사에서 내려오는 여러 형식의 지시서 문서에서 펀드 설정/해지 주문 데이터를 추출해 JSON/CSV로 저장한다.

지원 문서 형식:
- PDF
- Excel (`.xlsx`, `.xlsm`, `.xls`)
- HTML
- EML
- MHT

핵심 출력 데이터:
- `fund_code`
- `fund_name`
- `settle_class`
- `order_type`
- `base_date`
- `t_day`
- `transfer_amount`

즉, 문서 형식이 달라도 최종적으로는 같은 주문 스키마로 정규화하는 것이 목표다.

## 3. 전체 실행 흐름

전체 실행 흐름은 아래와 같다.

```text
main.py
  -> ExtractionComponent
    -> handler A
      -> DocumentLoader
        -> 형식별 로더(pdf/html/excel/eml/mht)
        -> raw_text 생성
        -> markdown_text 생성
        -> coverage 추정
        -> DocumentLoadTaskPayload JSON 저장
    -> handler B
      -> DocumentLoadTaskPayload JSON 재로딩
      -> FundOrderExtractor
        -> 7-stage LLM 추출
        -> 필요 시 base_date 이후 shortcut 복구
        -> 코드 기반 후처리/정규화
        -> 결과 검증
          -> blocking issue 검사
          -> coverage 검사
    -> JSON/CSV 저장
```

### 실제 호출 순서

1. `main.py`가 CLI 인자를 받는다.
2. `ExtractionComponent`가 단건/다건 실행 경로를 정리한다.
3. handler A가 `DocumentLoader`를 호출한다.
4. `DocumentLoader`가 문서를 읽어 `raw_text`, `markdown_text`, chunk, scope, coverage 메타데이터를 만든다.
5. handler A가 `DocumentLoadTaskPayload`를 JSON 파일로 저장한다.
6. handler B가 저장된 JSON 파일을 다시 읽는다.
7. `FundOrderExtractor`가 7-stage LLM 추출을 수행한다.
8. 필요하면 `base_date` 이후 구조 기반 shortcut/markdown shortcut으로 주문을 결정론적으로 복구한다.
9. extractor가 결과를 코드 규칙으로 다시 정리하고 coverage/blocking issue를 검사한다.
10. `ExtractionComponent`가 필요 시 merge/dedupe/CSV 평탄화를 적용한다.
11. 저장 가능한 결과면 JSON/CSV로 출력한다.

이때 handoff JSON 저장 위치와 삭제 여부는 env 설정으로 제어한다.
- `TASK_PAYLOAD_OUTPUT_DIR`
- `DELETE_TASK_PAYLOAD_FILES`

주의:
- 코드 기본값은 `DELETE_TASK_PAYLOAD_FILES=true`다.
- 현재 저장소의 `.env` 예시는 `false`로 잡혀 있어 handoff 파일을 남긴다.

즉 "기본 동작"과 "현재 저장소 예시 설정"은 다를 수 있다.

## 4. 왜 `raw_text`와 `markdown_text`를 둘 다 유지하나

이 프로젝트에서 가장 중요한 설계 포인트 중 하나다.

### `raw_text`

원문을 가능한 한 그대로 보존한 텍스트다.

용도:
- coverage 계산
- 디버깅
- markdown 변환이 놓친 정보 보완
- 후처리에서 원문 evidence 재확인

### `markdown_text`

LLM이 읽기 쉽게 구조화한 텍스트다.

용도:
- 표 헤더/본문 관계 전달
- row/column 구조 전달
- section 구분 전달

즉,
- `raw_text`는 "원문 보존"
- `markdown_text`는 "모델 친화적 구조화"
역할을 맡는다.

## 5. `DocumentLoader`가 하는 일

[app/document_loader.py](/Users/bhkim/Documents/codex_prj_sam_asset/app/document_loader.py)는 이 프로젝트의 전처리 중심축이다.

주요 책임:
- 파일 확장자별 로더 선택
- `raw_text -> markdown_text` 변환
- 주문 버킷 수 추정
- 삼성 운용사 대상 scope 계산
- 비지시서 문서 조기 판별

여기서 중요한 점은 이 파일이 단순 문자열 유틸이 아니라는 것이다.  
실제로는 "LLM이 오해하기 쉬운 문서를 코드로 먼저 정리하는 계층"이다.

### 자주 보게 되는 함수

- `load()`
  - 문서 형식에 맞는 로더를 호출한다.
- `build_markdown()`
  - raw text를 LLM용 markdown으로 바꾼다.
- `estimate_order_cell_count()`
  - 원문 기준 기대 주문 건수를 계산한다.
- `extract_target_fund_scope()`
  - 삼성 운용사 필터링 범위를 계산한다.

### 왜 heuristic이 많은가

문서 레이아웃이 매우 다양하기 때문이다.

예:
- split-row PDF
- rowspan HTML
- 운용사 헤더만 있는 문서
- `입금소계/출금소계`가 따로 있는 원장형 표
- plain-text로만 복원되는 PDF

이런 문서들은 정규식 한두 개로는 안정적으로 처리하기 어렵다.  
그래서 `document_loader.py`에는 "과대 계상/과소 계상/헤더 오염"을 막는 heuristic이 많이 들어 있다.

## 7. `FundOrderExtractor`가 하는 일

[app/extractor.py](/Users/bhkim/Documents/codex_prj_sam_asset/app/extractor.py)는 LLM 추출의 중심이다.

이 파일은 크게 3부분으로 나눠 읽으면 좋다.

### 1) Stage 입력/응답 모델

상단의 `FundSeedItem`, `FundSlotItem`, `FundResolvedItem` 같은 모델들이다.

역할:
- 각 단계의 JSON 계약 정의
- 응답 유효성 검증
- 불필요한 필드 무시

### 2) 7-stage LLM 실행

`FundOrderExtractor.extract()`가 시작점이다.

단계:
1. `fund_inventory`
2. `base_date`
3. `t_day`
4. `transfer_amount`
5. `settle_class`
6. `order_type`

왜 나누는가:
- 한 번에 다 뽑으면 필드 간 혼선이 커짐
- 단계별로 좁은 JSON 계약을 검증하는 편이 안정적임

### 3) 코드 기반 후처리

실제 도메인 정합성은 여기서 맞춘다.

대표 함수:
- `_build_result()`
- `_aggregate_candidates()`

또한 최종 출력 직전에는 두 가지 후처리가 더 있다.

- `only_pending`
  - `True`일 때 기존 `PENDING` 주문을 제거하고, 남은 주문의 `settle_class`를 `PENDING`으로 맞춤
- 출력 계약 치환
  - `settle_class`: `PENDING -> "1"`, `CONFIRMED -> "2"`
  - `order_type`: `RED -> "1"`, `SUB -> "3"`

이 둘은 내부 모델이 아니라 외부 JSON/CSV 계약에서만 적용된다.
- `_select_group_transfer_amount()`
- `_reconcile_final_issues()`

여기서 하는 일:
- 금액 정규화
- SUB/RED 방향 정규화
- 문서 내부 버킷 정리
- `결제금액`, `순유입금액`, `정산액` 같은 net 컬럼 우선 사용
- structured markdown fallback으로 coverage 복구
- 잘못 남은 issue 정리

즉, LLM은 후보를 만들고, 최종 책임은 코드가 진다.

## 8. coverage 검증이 중요한 이유

coverage는 "원문에 보이는 주문 버킷 수"와 "최종 추출 건수"가 크게 어긋나는지를 보는 안전장치다.

이 프로젝트에서는 LLM이 때때로:
- 일부 주문을 누락하거나
- 소계/합계 행을 주문으로 오인하거나
- 같은 주문을 둘로 나누는
문제가 생길 수 있다.

이걸 잡기 위해 `DocumentLoader`가 원문 기준 기대 버킷 수를 계산하고,  
현재 구현에서는 `FundOrderExtractor.extract_from_task_payload()`가 payload의 `expected_order_count`와 최종 결과를 비교한다.

즉 coverage는 정확도 그 자체가 아니라, **정확도가 무너졌을 때 빨리 잡는 안전장치**다.
