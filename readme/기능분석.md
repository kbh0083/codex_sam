- `transfer_amount` final payload는 소수 2자리 고정 출력이 아니라 source-scale 보존 정책을 따른다.
- 테스트 재수행 절차와 보고서 필수 항목은 [test_report/20260420_로컬_WAS_테스트_수행_방안.md](/Users/bhkim/Documents/codex_prj_sam_asset/test_report/20260420_로컬_WAS_테스트_수행_방안.md)를 따른다.

## 1. 시스템 개요

### 1.1 목적

이 애플리케이션은 **변액일임 설정/해지 지시서**에서 거래 주문 데이터를 자동으로 추출합니다.

### 1.2 지원 입력 형식

| 형식 | 확장자 | 처리 라이브러리 |
|------|--------|-----------------|
| PDF | `.pdf` | pdfplumber |
| HTML | `.html`, `.htm` | 내장 HTML text/table parser |
| MHT | `.mht`, `.mhtml` | MIME parser + HTML parser |
| EML | `.eml` | email.parser (내장), HTML 본문 추출 |
| Excel (신규) | `.xlsx`, `.xlsm` | openpyxl |
| Excel (구형) | `.xls` | xlrd |

### 1.3 추출 목표 데이터

각 주문(Order)은 다음 필드를 가집니다:

| 필드 | 설명 | 예시 |
|------|------|------|
| `fund_code` | 펀드 코드 | `"4070003"` |
| `fund_name` | 펀드명 | `"(무)푸르덴셜VUL롱텀밸류혼합형_주식2"` |
| `settle_class` | 결제 구분 | `CONFIRMED` / `PENDING` |
| `order_type` | 거래 유형 | `SUB`(설정) / `RED`(해지) |
| `base_date` | 기준일 | `"2025-08-26"` |
| `t_day` | 결제일(T+N) | `0`, `1`, `2` |
| `transfer_amount` | 이체 금액 | `"4260262"`, `"-18053366"` |

주의:
- 위 표는 **extractor 내부 모델** 기준이다.
- 실제 JSON/CSV 최종 출력은 아래 1.4 절의 문자열 코드 계약으로 한 번 더 직렬화된다.

### 1.4 출력 형태

- **CLI**: JSON 파일 + CSV 파일
- **외부 서버 연동**: `ExtractionComponent`를 통해 기존 WAS/배치 코드에서 동일한 추출/병합 규칙 재사용
- **단건 service 호출**: `ExtractionService`도 내부적으로는 handoff JSON 저장/재로딩 경로 사용

최종 외부 출력 계약은 내부 enum 문자열을 그대로 노출하지 않는다.

| 필드 | 내부 값 | 최종 출력 값 |
|------|---------|--------------|
| `settle_class` | `PENDING` | `"1"` |
| `settle_class` | `CONFIRMED` | `"2"` |
| `order_type` | `RED` | `"1"` |
| `order_type` | `SUB` | `"3"` |

또한 `only_pending=true`일 때는 최종 출력 직전에만 아래 규칙을 적용한다.
- 기존 `settle_class == PENDING` 주문 제거
- 남은 주문의 `settle_class`를 내부적으로 `PENDING`으로 맞춘 뒤 출력 코드 `"1"`로 저장

### 1.5 아키텍처 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│  외부 연동 경계         →  코드 (ExtractionComponent / WAS handler) │
│  handler A (로딩)      →  코드 (DocumentLoader)                    │
│  handler B (추출)      →  코드 (FundOrderExtractor)                │
│  중간 전달 계약         →  코드 (DocumentLoadTaskPayload JSON)      │
│  단건 호환 service      →  코드 (ExtractionService, 내부 handoff)   │
└─────────────────────────────────────────────────────────────────┘
```

- **전처리와 검증**: 결정론적(코드)
- **업무 의미 해석**: LLM 중심

## 3. 진입점 (Entry Points)

### 3.1 CLI (`main.py`)

로컬 검증, 배치 실행, 회귀 테스트용입니다.

**실행 예시:**
```bash
# 기본 실행 (PDF/HTML/EML/Excel)
python main.py -f /path/to/order.pdf
python main.py -f /path/to/order.html
python main.py -f /path/to/order.eml

# 암호화 PDF
python main.py -f /path/to/locked.pdf -p your-password

# 출력 경로 지정
python main.py -f /path/to/order.pdf -o /path/to/result
```

**실행 단계:**
```
[1/7] 입력 파일 경로 해석
  ↓
[2/7] ExtractionComponent 생성 및 출력 경로 준비
  ↓
[3/7] DocumentExtractionRequest 목록 생성
  ↓
[4/7] handler A/B 기반 추출 실행
  ↓
[5/7] 추출 완료 로그
  ↓
[6/7] JSON 저장
  ↓
[7/7] CSV 저장
```

**특징:**
- `ExtractionComponent`가 반환한 payload를 그대로 JSON/CSV로 저장한다.
- 문서 상태는 `COMPLETED`, `SKIPPED`, `FAILED`로 직렬화될 수 있다.
- 파일 경로 해석 실패 같은 CLI 경계 예외일 때만 nonzero 종료한다.
- 다건 실행이어도 내부 merge 규칙은 `ExtractionComponent`가 단건과 같은 주문 스키마를 유지한 채 처리

### 3.2 외부 서버 연동

현재 저장소에는 FastAPI 엔트리가 없습니다.  
기존 개발 서버나 별도 WAS에서 추출 기능을 재사용할 때는 `ExtractionComponent`를 권장 진입점으로 사용합니다.
이 컴포넌트가 단건 추출, 다건 merge, CSV 평탄화 규칙을 한 번에 감싸기 때문입니다.

```python
from app.component import DocumentExtractionRequest, ExtractionComponent

component = ExtractionComponent()
payload = component.extract_document_payload(
    DocumentExtractionRequest(
        source_path="/path/to/order.pdf",
        pdf_password=None,
    )
)
```

## 5. 데이터 추출 프로세스 - 함수 호출 상세 흐름

> 처음 소스를 접하는 개발자가 `main.py`에서 시작해 전체 파이프라인을 따라갈 수 있도록, **함수 호출 단계**를 상세히 정리합니다.

### 5.1 전체 호출 트리 (main → 최종 출력)

```
main()                                    [main.py]
├── setup_logging()                        [app/logging_utils.py]
├── build_parser().parse_args()            [main.py]
├── ExtractionComponent()                  [app/component.py]
│   ├── get_settings()                     [app/config.py]
│   ├── DocumentLoader()                   [app/document_loader.py]
│   └── FundOrderExtractor(settings)       [app/extractor.py]
├── build_output_paths()                  [main.py]
└── component.extract_document_payload(s)  [app/component.py]
    ├── run_handler_a()                     [app/component.py]
    │   ├── _validate_source_path()         [app/component.py]
    │   ├── loader.build_task_payload()     [app/document_loader.py]
    │   └── DocumentLoadTaskPayload.write_json()
    ├── run_handler_b()                     [app/component.py]
    │   ├── DocumentLoadTaskPayload.read_json()
    │   ├── extractor.extract_from_task_payload() [app/extractor.py]
    │   └── write_invalid_response_debug_files()
    └── (다건인 경우) build_merged_payload() [app/component.py]
```

### 5.2 단계별 함수 호출 상세

#### Phase 0: 초기화 (main.py / component.py)

| 순서 | 호출 위치 | 함수 | 파일 | 설명 |
|------|-----------|------|------|------|
| 1 | `main()` | `setup_logging()` | `app/logging_utils.py` | 로깅 포맷 설정 |
| 2 | `main()` | `build_parser().parse_args()` | `main.py` | CLI 인자 파싱 |
| 3 | `main()` | `ExtractionComponent()` | `app/component.py` | handler A/B를 직접 수행하는 상위 파사드 생성 |

**ExtractionComponent.__init__() 내부:**
- `get_settings()` → `.env` 로드
- `DocumentLoader()` → 로더 인스턴스 생성
- `FundOrderExtractor(settings)` → 추출기 생성

**참고:**  
`ExtractionService`를 직접 쓰는 경우에도 내부적으로는 같은 handoff 파일 경로를 거친다.  
즉 service는 현재 "단건 호환용 래퍼"이면서도 실행 방식은 component와 최대한 같게 맞춰져 있다.

---

#### Phase 1: handler A (DocumentLoader 중심)

| 순서 | 호출 위치 | 함수 | 파일 | 설명 |
|------|-----------|------|------|------|
| 1 | `extract_document_payload()` | `run_handler_a()` | `app/component.py` | 문서 1건의 로딩 단계 수행 |
| 2 | `run_handler_a()` | `_validate_source_path()` | `app/component.py` | 입력 경로를 실제 문서 경로로 해석 |
| 3 | `run_handler_a()` | `loader.build_task_payload()` | `app/document_loader.py` | raw/markdown/chunk/scope/coverage 메타데이터 생성 |
| 4 | `run_handler_a()` | `DocumentLoadTaskPayload.write_json()` | `app/document_loader.py` | handler A 산출물 JSON 저장 |

`DocumentLoader.build_task_payload()` 안에서는 아래가 함께 처리된다.
- 형식별 원문 로드
- `raw_text` 생성
- `markdown_text` 생성
- `chunks` 생성
- 비지시서 판정 정보 계산
- `expected_order_count` 계산
- `target_fund_scope` 계산

즉 handler A는 "문서를 읽어서 다음 단계가 다시 파일을 열지 않아도 되게 만드는 곳"이다.

---

#### Phase 2: handler B (FundOrderExtractor 중심)

| 순서 | 호출 위치 | 함수 | 파일 | 설명 |
|------|-----------|------|------|------|
| 1 | `extract_document_payload()` | `run_handler_b()` | `app/component.py` | 추출 단계 수행 |
| 2 | `run_handler_b()` | `DocumentLoadTaskPayload.read_json()` | `app/document_loader.py` | handler A 산출물 복원 |
| 3 | `run_handler_b()` | `extractor.extract_from_task_payload()` | `app/extractor.py` | 7-stage LLM 추출 + 후처리 + 검증 |
| 4 | `run_handler_b()` | `write_invalid_response_debug_files()` | `app/extractor.py` | invalid raw response artifact 저장 |

`extract_from_task_payload()`가 하는 일:
- `allow_empty_result` 규칙 반영
- `target_fund_scope` 반영
- 7-stage LLM 추출 실행
- coverage mismatch 검사
- blocking issue 검사
- 실패 시 `ExtractionOutcomeError`로 partial result + invalid artifact 전달

---

#### Phase 3: 다건 merge / 출력

| 순서 | 호출 위치 | 함수 | 파일 | 설명 |
|------|-----------|------|------|------|
| 1 | `extract_document_payloads()` | 문서별 handler A/B 반복 | `app/component.py` | 입력 배열이라도 문서별 개별 추출 수행 |
| 2 | `extract_merged_payload()` | `build_merged_payload()` | `app/component.py` | 다건 merge 적용 |
| 3 | `main()` | `json_path.write_text()` | `main.py` | JSON 저장 |
| 4 | `main()` | `write_orders_csv()` | `main.py` | CSV 저장 |

다건 merge는 "문서를 한 번에 합쳐 추출"하는 것이 아니라,
각 문서를 먼저 독립적으로 추출하고 마지막에만 병합 규칙을 적용한다.

---

#### Phase 4: service 직접 호출 경로

`ExtractionService.extract_file_path()`를 직접 호출할 때도 아래 순서를 따른다.

1. `_build_task_payload()`로 `DocumentLoadTaskPayload` 생성
2. `_reload_task_payload_via_handoff_file()`로 JSON 저장/재로딩
3. `FundOrderExtractor.extract_from_task_payload()` 호출
4. invalid artifact 저장
5. 결과 반환

즉 service도 더 이상 "로더 결과를 메모리에서 바로 extractor로 넘기는 별도 경로"가 아니다.

---

#### Phase 5: 출력 (main.py)

| 순서 | 호출 위치 | 함수 | 파일 | 설명 |

### 6.2 DocumentLoader (`app/document_loader.py`)

`DocumentLoader`는 이제 형식별 로더를 직접 구현하지 않고 조합합니다.

- PDF 추출: `app/document_loaders/pdf_loader.py`
- HTML 추출: `app/document_loaders/html_loader.py`
- EML 추출: `app/document_loaders/eml_loader.py`
- Excel 추출: `app/document_loaders/excel_loader.py`

**역할**: LLM 친화적 문서 정규화기 (단순 파일 로더가 아님)

**출력 구조 `ExtractedDocumentText`:**
```python
ExtractedDocumentText(
    raw_text="원문 텍스트 (검수/coverage 기준)",
    markdown_text="LLM 입력용 markdown",
    content_type="application/pdf" | "application/vnd.ms-excel" | ...
)
```

**주요 기능:**

1. **PDF 처리** (`_load_pdf`)
   - 여러 `extract_tables()` 설정 시도 → 가장 덜 fragmented 후보 선택
   - preamble / header / body 분리
   - `2. Redemption` 같은 섹션 마커를 문맥 텍스트로 분리
   - 멀티페이지 표: 이전 페이지 header carry-over
   - 표 외부 문맥 라인 병합 (Subscription, Redemption, 기준일자 등)
   - pipe-delimited raw text로 정규화

2. **HTML 처리** (`_load_html`)
   - 내장 정규식 기반 파서 (BeautifulSoup 등 외부 의존 없음)
   - `<body>` 내부 추출 후 테이블/텍스트 분리
   - `_expand_html_table()`: rowspan/colspan 전개 → pipe-delimited 행
   - `[HTML 파일명]\ncol1 | col2 | col3\n...` 형식
   - `_format_markdown_heading()`: `[HTML ...]` → `## HTML ...` 섹션 헤딩

3. **EML 처리** (`_load_eml`)
   - `email.parser.BytesParser`로 메일 파싱
   - multipart에서 `text/html` 또는 `text/plain` 본문 추출
   - 이전 메시지/quoted thread가 붙어 있으면 현재 메시지 경계까지만 남기고 이후 내용은 버림
   - HTML 본문은 HtmlDocumentLoaderMixin의 `_extract_html_raw_text()`로 변환
   - `_score_eml_body_lines()`: 주문서 키워드, pipe 행, 날짜 등으로 품질 점수 → 최적 후보 선택
   - `[EML 파일명]\nSubject: ...\n...` 형식, content_type: `message/rfc822`

4. **Excel 처리**
   - `.xlsx`, `.xlsm`: openpyxl
   - `.xls`: xlrd
   - 시트별 `[SHEET 시트명]\ncol1 | col2 | col3\n...` 형식

5. **Markdown 생성** (`build_markdown`)
   - `[PAGE N]`, `[SHEET name]`, `[HTML filename]`, `[EML filename]` → section heading
   - pipe row block → markdown table
   - 일반 텍스트 → fenced text block

6. **Chunk 분할** (`split_for_llm`)
   - section delimiter 기준으로 분할 (임의 절단 방지)
   - `chunk_size_chars` 이내로 유지

7. **Coverage 추정** (`estimate_order_cell_count`)
   - explicit order: 설정금액, 해지금액, 입금액, 출금액 등
   - execution: 당일이체금액, 실행금액
   - scheduled: 날짜/T+N 컬럼
   - `target_fund_scope` 전달 시 운용사 필터 반영하여 버킷 수 추정
   - 이 값과 최종 order 개수 비교

8. **운용사 필터링** (`extract_target_fund_scope`)
   - 원문에 운용사(운용사명, manager) 컬럼/헤더가 있으면 삼성 계열 펀드만 추출 대상
   - `TargetFundScope`: manager_column_present, include_all_funds, fund_codes, fund_names

## 7. 7-stage LLM 추출 파이프라인

### Stage 1: instruction_document

| 항목 | 내용 |
|------|------|
| **입력** | chunk 단위 document text |
| **출력** | `is_instruction_document`, `reason` |
| **목적** | 실제 주문 행이 있는 지시서인지 먼저 판정 |

### Stage 2: fund_inventory

| 항목 | 내용 |
|------|------|
| **입력** | chunk 단위 document text |
| **출력** | fund_code, fund_name |
| **목적** | 문서에 등장한 펀드의 기준 집합 확보 |
| **운용사 필터** | 운용사/운용사명/manager 컬럼이 있으면 삼성 포함 펀드만 추출, 없으면 전체 |

### Stage 3: base_date

| 항목 | 내용 |
|------|------|
| **입력** | Stage 2 결과 |
| **출력** | 각 fund의 base_date |
| **목적** | 기준일 해석 (기준일, 기준일자, T일, 결제일 등) |

### Stage 4: t_day

| 항목 | 내용 |
|------|------|
| **입력** | Stage 3 결과 |
| **출력** | t_day, slot_id, evidence_label |
| **목적** | 주문 슬롯 식별 |

**slot_id 예시:** `T0_1`, `T1_1`, `T1_2`  
**evidence_label 예시:** `입금액`, `출금액`, `당일이체금액`, `2025-11-28`, `Redemption`

### Stage 5: transfer_amount

| 항목 | 내용 |
|------|------|
| **입력** | Stage 4 결과 |
| **출력** | transfer_amount |
| **규칙** | 문서/추출 문자열에 실제 소수 금액이 있으면 소수 자릿수까지 유지하고, spreadsheet float-tail artifact는 nearest integer amount로 정규화. 좌수/unit/share 계열은 금지 |

금액 출력 예시는 아래와 같다.

- `23,213.40 -> 23,213.40`
- `23,213.4 -> 23,213.4`
- `23,213.409 -> 23,213.409`
- `18,711,858.00 -> 18,711,858`
- `70,000,000.00000001 -> 70,000,000`

backend final payload 단계에서는 소수 2자리 padding/truncation을 하지 않는다.

### Stage 6: settle_class

| 항목 | 내용 |
|------|------|
| **입력** | Stage 5 결과 |
| **출력** | `CONFIRMED` / `PENDING` (내부 모델 값) |

### Stage 7: order_type

| 항목 | 내용 |
|------|------|
| **입력** | Stage 6 결과 |
| **출력** | `SUB` / `RED` (내부 모델 값) |
| **판단 우선순위** | 1) 금액 부호 2) evidence/header 단서 3) LLM 출력값 |

### Stage 실행 방식

- **Stage 1~2**: chunk별로 독립 실행 → 결과 병합
- **Stage 3~7**: `stage_batch_size`(기본 12) 단위로 배치 실행
- 각 stage: system prompt + user prompt (목표, plan, 규칙, JSON contract, 입력 item, 문서 text)

### base_date 이후 shortcut 경로

2026-04-16 기준 현재 extractor는 `base_date`가 정리된 뒤 아래 shortcut을 먼저 시도한다.

1. **structure shortcut**
   - 표/섹션 구조만으로 주문을 결정론적으로 조립할 수 있으면 이후 stage 일부를 생략한다.
   - 흥국생명처럼 `fund_name`이 비어 있고 `fund_code + 금액`만 남는 문서도 이 경로에서 복구한다.
2. **markdown shortcut**
   - markdown table이 균일하면 planned/confirmed 슬롯과 signed amount를 코드 기반으로 조립한다.
3. **일반 stage 계속 실행**
   - shortcut으로 충분하지 않을 때만 나머지 LLM stage를 수행한다.

즉 현재 구현은 "항상 stage 1~7 전체 실행"이 아니라, `base_date` 이후 결정론적 복구를 먼저 시도하는 형태다.

### 최종 order 생성 (`_build_result`)

1. fund_code, fund_name 정리
2. evidence가 좌수/unit/share면 제거
3. base_date, transfer_amount, settle_class, t_day, order_type 정규화
4. 0, -, 빈칸 금액 제거
5. explicit vs execution conflict 해소 (`_select_group_transfer_amount`, `_aggregate_candidates`)

### explicit vs execution conflict

동일 행에 다음이 함께 있을 수 있음:
- **explicit**: 설정금액, 해지금액, 입금액, 출금액
- **execution/net**: 결제금액, 순유입금액, 순투입금액, 정산액, 당일이체금액, 실행금액, 이체예정금액

현재 규칙은 다음과 같다.
- 같은 버킷에 execution/net 컬럼이 있으면 그 값을 우선 사용한다.
- 같은 버킷에 explicit inflow/outflow만 있고 execution/net 컬럼이 없으면 `SUB`와 `RED`를 분리 유지한다.
- 예: `동양생명_20260318.html`의 `정산내역`은 `정산액`을 transfer_amount와 방향 판단 기준으로 사용한다.

즉 stage에서는 후보를 폭넓게 허용하고,
최종적으로는 `_build_result()`에서 후보 생성 후 `_aggregate_candidates()` → `_select_group_transfer_amount()`로 버킷별 규칙을 정리한다.

### 8.1 Coverage 검증

```
expected_order_count = loader.estimate_order_cell_count(raw_text, target_fund_scope)
actual_count = len(extraction_result.orders)

expected_order_count != actual_count
  → 우선 `ORDER_COVERAGE_MISMATCH` 이슈 추가
  → 독립 deterministic corroboration이 있으면 `ORDER_COVERAGE_ESTIMATE_MISMATCH` soft warning으로 완화 가능
  → 현재 구현에서는 coverage mismatch 단독으로는 hard blocker가 아님
```

**의미:** stage 누락, PDF 표 붕괴, 헤더 오해석, 중복 제거 과잉 등을 조기에 발견.

### 10.1 Invalid Response 디버그 파일

LLM stage 응답이 JSON 규약을 만족하지 못하면 raw 응답을 파일로 저장합니다.

- **위치**: `DEBUG_OUTPUT_DIR`
- **파일명**: `{문서명}_{timestamp}_{stage}_chunk{NN}_llm_invalid_response.txt`

### 10.2 로그 읽기 순서

1. 파일 로드
2. markdown 생성
3. chunk 분할
4. Stage 1~7
5. coverage 검증
6. 저장 또는 차단

### 10.3 코드 읽기 권장 순서

1. `main.py` - CLI 흐름
2. **[§5 데이터 추출 프로세스 - 함수 호출 상세 흐름](#5-데이터-추출-프로세스---함수-호출-상세-흐름)** - 전체 호출 트리 확인
3. `app/service.py` - 오케스트레이션
4. `app/document_loader.py` - 문서 정규화
5. `app/extractor.py` - 7-stage 추출
6. `app/schemas.py` - 데이터 계약
